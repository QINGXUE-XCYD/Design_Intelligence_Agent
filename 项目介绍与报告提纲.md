# 项目介绍与报告提纲

## 1. 项目简介

本项目是一个基于 Python 的二维栅格仿真系统，用于研究多机器人在部分未知室内环境中的自主探索、地图构建、路径规划、协同清扫与资源竞争问题。项目面向 `Designing Intelligent Agents / Intelligent Agents` 课程作业，核心目标是构建一个可重复运行、可配置、可比较、可统计分析的智能体实验平台。

当前项目已经完成了较完整的主线：

- 环境建模
- 机器人局部感知
- belief map 更新
- frontier 探索
- A* 路径规划
- 多机器人协作
- 电量与充电机制
- 指标统计
- 四组正式对比实验

与之前不同的是，项目现在已经清理掉早期单次演示和旧可视化入口，代码结构明确聚焦于“仿真环境主链 + 正式实验脚本”。

## 2. 项目研究问题

本项目主要围绕以下四类问题展开：

1. 机器人数量变化会如何影响清扫效率与重复劳动？
2. 不同协作策略会如何影响系统整体表现？
3. 不同感知能力和感知模式会如何影响任务完成质量与效率？
4. 在存在电量限制的情况下，充电资源竞争是否会成为系统瓶颈？

这四个问题正好对应当前已经完成的四组正式实验，也适合你们 4 人小组按“一人负责一个 question”的方式分工。

## 3. 系统整体实现思路

### 3.1 环境建模

真实环境由 `environment/grid_map.py` 和 `environment/map_generator.py` 管理。

地图采用二维栅格表示，生成逻辑包括：

- 在边界处添加墙体
- 在内部随机放置静态障碍
- 每放置一个障碍后进行一次连通性检查

这样做的目的是避免生成大量封闭死区，保证大部分自由空间仍然位于同一连通区域中，从而使实验环境既具有随机性，又尽量避免无效地图。

系统还支持动态障碍扩展，但当前正式实验默认关闭动态障碍。

### 3.2 机器人内部状态与行为逻辑

机器人核心逻辑在 `agents/robot_agent.py`，状态定义在 `agents/agent_state.py`。

每个机器人维护如下信息：

- 当前坐标
- 当前目标
- 当前路径
- 运行模式
- 轨迹记录
- 已清扫格子数量
- 电量状态
- 已知充电站信息

在每一个仿真 step 中，机器人主要执行如下过程：

1. 若在充电站且需要充电，则优先充电或等待充电
2. 感知局部环境
3. 更新自己的 belief map
4. 清扫当前位置
5. 检查当前目标是否需要重新规划
6. 若需要，则重新选择目标
7. 规划到目标的路径
8. 执行动作并移动
9. 更新状态与统计量

这体现了一个典型智能体流程：感知、内部表示更新、决策、规划、执行。

### 3.3 感知模块

感知逻辑实现于 `sensing/sensor_model.py`。

当前支持三种感知模式：

- `manhattan`
  - 默认模式
  - 在固定曼哈顿距离范围内可见
  - 不考虑遮挡

- `euclidean`
  - 使用圆形感知半径
  - 不考虑遮挡

- `occluded_manhattan`
  - 曼哈顿距离范围限制
  - 加入视线遮挡
  - 若中间有障碍，则障碍后方格子不可见

此外，底层还支持感知噪声：

- `false_positive_rate`
- `false_negative_rate`

不过当前正式实验并未将噪声作为主线，因为现有噪声设置下系统性能下降过于剧烈，不利于形成稳定、清晰的对比结论。

### 3.4 地图表示与地图融合

机器人的内部地图使用 occupancy grid 表示，定义在 `mapping/occupancy_grid.py`。

每个格子可处于三种状态：

- `UNKNOWN`
- `FREE`
- `OCCUPIED`

同时还单独记录格子是否已被清扫。

在多机器人协作场景中，若策略包含 `shared_map`，系统会在 `simulation/engine.py` 中维护一个共享地图，并定期执行双向通信：

1. 每个机器人将本地地图上传到 shared map
2. shared map 再下发回各机器人

当前融合规则是保守型的：

- `OCCUPIED` 永远覆盖
- `FREE` 只覆盖 `UNKNOWN`
- `cleaned` 状态按逻辑或合并

因此，这里的地图融合本质上是“同坐标系下的共享 occupancy grid 合并”，并不是高级 SLAM，也不涉及地图对齐。

### 3.5 frontier 探索

探索逻辑实现于 `exploration/frontier_detector.py`。

frontier 的定义是：

- 当前格子已知且可通行
- 邻居中至少存在一个未知格子

机器人会优先前往 frontier，从而逐步扩展已知地图。这是经典的部分未知环境探索策略。

### 3.6 路径规划

路径规划在 `planning/astar_planner.py` 中实现，使用 A* 算法。

其特点包括：

- 启发函数为曼哈顿距离
- 默认只在 belief map 中“已知可通行”的格子上规划
- 若目标不可达，则返回空路径

因此机器人并不是盲目随机游走，而是通过“目标选择 + A* 规划”的方式完成有目的的探索与清扫。

### 3.7 协作策略

当前支持四种协作策略，定义于 `config/schema.py`，逻辑主要在 `simulation/engine.py` 和 `agents/robot_agent.py` 中：

- `independent`
  - 每个机器人只使用自己的本地地图
  - 不共享信息，也不避免目标冲突

- `shared_map`
  - 共享地图和已发现的充电站信息
  - 但不避免选中同一目标

- `goal_reservation`
  - 不共享地图
  - 但尽量避免多个机器人选择同一目标

- `shared_map_reservation`
  - 同时共享地图和进行目标预留
  - 是当前最完整的协作模式

这组策略非常适合作为对比实验，因为它们分别体现了“信息共享”和“目标冲突避免”两个关键维度。

### 3.8 电量与充电机制

电量逻辑主要实现于 `agents/robot_agent.py` 和 `simulation/factory.py`。

当前设计包括：

- 每个机器人具有电池容量
- 移动、感知、清扫和通信都会消耗能量
- 当电量较低时，机器人会主动返航充电
- 机器人已知的充电站包括：
  - 自己的 home charger
  - 感知发现的公共充电站
  - 或由 shared map 共享而来的充电站

充电机制还有两个关键点：

- 同一充电站允许容量限制 `charging_station_capacity`
- 若同一步中站上的机器人数量超过容量，则部分机器人进入 `WAITING_FOR_CHARGE`

这使得系统不仅能研究清扫问题，还能研究资源竞争问题。

### 3.9 仿真引擎与终止条件

仿真主循环在 `simulation/engine.py`。

每一个 step 中，引擎负责：

- 推进动态障碍
- 执行地图通信
- 处理充电权限分配
- 依次让每个机器人执行一步
- 记录全局指标

终止条件包括：

- 达到最大步数
- 达到目标覆盖率
- 所有机器人都进入 `DONE`

### 3.10 指标统计与结果保存

指标统计位于 `metrics/collector.py`。

当前系统会记录：

- `coverage_rate`
- `total_steps`
- `total_path_length`
- `duplicate_visit_count`
- `inter_agent_overlap_cells`
- `steps_to_80_coverage`
- `steps_to_90_coverage`
- `steps_to_95_coverage`
- `total_energy_used`
- `total_charging_steps`
- `total_charging_events`
- `total_charge_wait_steps`
- `total_low_battery_returns`
- `total_battery_budget_returns`

其中几个关键指标含义如下：

- `duplicate_visit_count`
  - 所有机器人轨迹合并后产生的重复访问次数

- `inter_agent_overlap_cells`
  - 被多个机器人访问过的公共格子数

- `total_charge_wait_steps`
  - 机器人到达充电站后因为容量不足而等待的总步数

- `total_charging_events`
  - 进入充电状态的总次数
  - 它也能反映充电过程是否被打断得较碎

当前结果保存工具集中在 `utils/run_saver.py`，主要服务于批量实验输出，而不再负责早期单次演示文件。

## 5. 实验总体设计思路

实验设计遵循统一逻辑：

- 固定大部分系统参数
- 只改变某一个核心因素
- 观察系统行为是否发生系统性变化

实验统一采用如下方法：

- 固定地图大小、障碍密度和目标覆盖率
- 每组条件运行多个随机种子
- 统计均值、标准差和成功率
- 生成可直接用于报告的图表
- 再结合结果进行分析

这种设计满足课程要求中“run multiple times, compare conditions, use statistics and charts”的基本规范。

## 6. 四组正式实验说明

### 6.1 实验一：机器人数量对效率的影响

脚本：`experiments/exp1_robot_count.py`

实验目的：

- 研究机器人数量增加后系统整体效率是否提升
- 观察多机器人是否带来更多重复访问或冲突

- `num_agents = [1, 2, 3, 4]`

控制变量：

- 协作策略固定为 `shared_map_reservation`
- 地图大小、障碍密度、感知范围固定
- 动态障碍关闭

主要指标：

- `coverage_rate`
- `total_steps`
- `steps_to_80_coverage / 90 / 95`
- `duplicate_visit_count`
- `inter_agent_overlap_cells`
- `total_energy_used`

自变量：

- `independent`

- `goal_reservation`
- `shared_map_reservation`

控制变量：

- `num_agents = 3`
- 感知范围、地图大小、障碍密度固定

主要指标：

### 6.3 实验三：感知能力与感知模式的影响

脚本：`experiments/exp3_sensing.py`

实验三分为两个部分。

第一部分是感知范围实验：

- `sensor_range = [2, 3, 4, 5]`

第二部分是感知模式实验：

- `manhattan`
- `euclidean`
- `occluded_manhattan`

实验目的：

- 研究感知范围变大后效率是否持续提升
- 比较不同感知模式是否会改变探索和路径选择效果

主要指标：

- `coverage_rate`
- `total_steps`
- `duplicate_visit_count`
- `steps_to_80_coverage / 90 / 95`
- `total_energy_used`

主要图表：

- 不同范围的覆盖率曲线
- 不同范围的平均总步数、重复访问、能耗图
- 不同感知模式的覆盖率曲线
- 不同感知模式的平均总步数、重复访问、能耗图

当前结果表明，`occluded_manhattan` 略优于默认 `manhattan`。一个合理解释是：在障碍较多的环境中，某些几何距离近的点实际需要绕路到达，透视式感知会让机器人过早关注这些目标，而带遮挡的感知反而让机器人优先处理真正可见且路径更合理的区域。

### 6.4 实验四：充电竞争的影响

脚本：`experiments/exp4_charging_competition.py`

实验目的：

- 研究在电量限制存在时，充电资源是否会成为系统瓶颈
- 分析机器人数量与充电站容量之间的关系

实验采用二维设计：

- `num_agents = [2, 3, 4]`

同时为了更容易触发充电行为，实验采用较严格的电池参数：

- `battery_capacity = 100.0`
- `low_battery_threshold = 22.0`
- `recharge_rate = 12.0`

控制变量：

- 协作策略固定为 `shared_map_reservation`
- 地图大小、障碍密度、感知范围固定
- 动态障碍关闭
- 关闭额外公共充电站

主要指标：

- `coverage_rate`
- `total_steps`
- `total_charging_steps`
- `total_charging_events`
- `total_charge_wait_steps`
- `success_rate`

主要图表：

- 等待充电步数热力图

## 8. 项目目前的完成情况与限制

### 8.1 已经比较完整的部分

- 环境、智能体、感知、地图、规划、协作、电量、指标、实验框架都已实现
- 已经完成四组正式对比实验
- 已有多组实验结果、图表和中文分析
- 项目结构较清晰，便于报告撰写

### 8.2 需要在报告中说明的限制

- 当前地图融合是基础 shared occupancy grid，不是高级 SLAM
- 当前所谓任务分配主要是目标预留，不是复杂的 auction 或区域分配算法
- 动态障碍支持存在，但不是本次正式实验主线
- 感知噪声功能存在，但本次没有纳入正式实验结果
- 当前一个充电站位置允许多个机器人同时站上去，容量控制的是“同时充电人数”，不是“格子占位互斥”

这些限制不会影响课程作业达标，但在报告中需要表达清楚。
